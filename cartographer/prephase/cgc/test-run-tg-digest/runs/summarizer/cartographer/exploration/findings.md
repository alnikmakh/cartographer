## Purpose

The summarizer package provides LLM-powered message summarization for a Telegram digest application. Callers hand it a store full of messages and get back per-message summaries — either generated by an LLM (Ollama or OpenRouter) or copied verbatim for short messages. The primary entry points process a queue of unsummarized messages within a rolling two-day window, making this the batch-processing engine that turns raw channel posts into digestible summaries.

**Patterns**

- **Strategy** — `Client` interface with two implementations (`OllamaClient`, `OpenRouterClient`), injected into `Service` at construction. Callers choose the LLM backend without `Service` knowing which.
- **Functional Options** — `ServiceOption` / `WithLocalModel()` for optional configuration.
- **Sequential Queue Processor** — `SummarizeUnsummarized` fetches all pending work, processes one-by-one, stops on first error.

**Flow 2: API error halts queue**

1. Steps 1-4 as above
2. `service.go:113` — `SummarizeMessage` returns error from LLM client
3. `service.go:114-116` — increments `APIErrors`, records `LastError`, `break`s out of loop
4. Remaining messages are not processed; `result.Total - result.Processed - result.APIErrors` messages are left untouched

## Non-Obvious Behaviors

- **150-character threshold skips the LLM entirely** (`service.go:45`). Messages with `len(text) < 150` are copied verbatim as their own summary and stored immediately. The threshold is on byte length (`len()`), not rune count — multibyte UTF-8 text hits the threshold sooner. The boundary is `>=` 150: exactly 150 bytes triggers the LLM call (confirmed by `TestService_SummarizeMessage_ExactThreshold`).

- **Queue stops on first API error** (`service.go:116`). A single LLM failure halts the entire queue via `break`. The error is recorded in `SummarizeQueueResult.LastError` but `SummarizeUnsummarized` itself returns `nil` error — the caller must inspect the result struct. Unprocessed messages remain unsummarized for the next run.

- **Channel lookup failures are silently skipped** (`service.go:107`). If `GetByID` returns an error or nil, that message is skipped with `continue` — no error recorded, no counter incremented. The message stays unsummarized and will be retried on the next run, but the `Total` count includes it, so `Total > Processed + APIErrors` signals silently skipped messages.

- **OllamaClient reuses OpenRouter's wire types** (`ollama.go:30-38`). The Ollama client converts `ChatRequest` → `openRouterRequest` (defined in `openrouter.go`) for JSON serialization, because Ollama exposes an OpenAI-compatible endpoint at `/v1/chat/completions`. This means both clients share the same JSON schema despite being different services.

- **OpenRouterClient has no timeout** (`openrouter.go:52`). `NewOpenRouterClient` creates an `http.Client{}` with zero timeout. `OllamaClient` accepts a `timeout` parameter. Context cancellation is the only timeout mechanism for OpenRouter calls.

- **Connection error detection is string-based** (`ollama.go:156-158`). `isConnectionError` checks `err.Error()` for substrings "refused", "connect:", "no such host". This is fragile and only used by `OllamaClient` to produce user-friendly "ollama is not running" messages.

- **Local model preamble is appended, not prepended** (`service.go:51`). `WithLocalModel()` appends extra instructions ("Keep your response very concise", "Do not add preamble") to the end of the system prompt via string concatenation. This gives the local-model instructions last-position emphasis in the prompt.

- **Time window is UTC-only and non-configurable** (`service.go:90-93`). The yesterday+today window is hardcoded to UTC midnight boundaries. There is no timezone or window-size parameter.
